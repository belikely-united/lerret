# CLI Reference

The Lerret CLI ships as the npm package **`@lerret/cli`**. It installs one binary, named `lerret`, with three subcommands: `dev`, `export`, and `clear`. There is no `init` subcommand — use [`create-lerret`](#create-lerret) to scaffold projects.

## Running the CLI

| Context | Command |
|---|---|
| Zero-install (`npx`) | `npx @lerret/cli@latest dev` |
| Installed as a project dep | `@lerret/cli dev` (via `package.json` scripts, `pnpm exec`, `npx`, etc.) |
| Installed globally | `@lerret/cli dev` |

> Don't run `npx lerret …` — the bare `lerret` name belongs to a different (unrelated, deprecated) npm package. Always include the `@lerret/cli` scope when invoking via `npx`.

The command syntax shown below uses the canonical `@lerret/cli` form — it works identically in every context above.

### Exit codes

Every subcommand uses the same convention:

| Code | Meaning |
|---|---|
| `0` | Success — work completed, `--help` printed, or graceful shutdown. |
| `1` | Failure — unknown subcommand, malformed flags, missing project, fatal runtime error. |

`@lerret/cli <command> --help` (or `-h`) prints command-specific help and exits with code `0`.

## `@lerret/cli dev`

Runs the studio against a `.lerret/` project folder. Starts a local Vite dev server, bundles the studio, and opens it in your browser. The server is long-running — it stays up until you `Ctrl+C` (SIGINT) or send SIGTERM.

```bash
@lerret/cli dev [options]
```

### Options

| Flag | Type | Default | Description |
|---|---|---|---|
| `--port <n>` | integer (1–65535) | Vite's default (usually 5173) | Dev-server port. |
| `--folder <path>` | path | (walk up from CWD) | Project folder. Bypasses the walk-up auto-detection. |
| `--open` | boolean | true | Open the studio in the default browser on start. |
| `--no-open` | boolean | (sets `open=false`) | Do not open the browser. |
| `-h, --help` | — | — | Print this help and exit `0`. |

Unknown flags are a usage error (exit `1`).

### Project resolution

`@lerret/cli dev` resolves the project in this order:

1. **`--folder <path>`** if supplied. The path is used directly; the CLI checks that it (or a parent) contains a `.lerret/` directory.
2. Otherwise the CLI **walks up** from the current working directory looking for the nearest `.lerret/` — the way `git` walks up looking for `.git/`.

If neither path finds a project, the command exits with code `1` and a clear message.

### Examples

```bash
@lerret/cli dev                          # walk up from cwd, default port, auto-open
@lerret/cli dev --port 4000              # custom port
@lerret/cli dev --no-open                # headless: useful in CI / SSH
@lerret/cli dev --folder ./design-tokens # explicit folder, skip walk-up
```

## `@lerret/cli export`

Headlessly renders every artboard in scope to image files. The CLI boots a Vite server, launches a headless Chromium through Playwright, navigates the studio against your project, and captures each artboard. Uses the **same rendering path** as the studio's per-artboard PNG button, so the output is pixel-faithful to the studio.

```bash
@lerret/cli export [path] [options]
```

### Argument

| Argument | Description |
|---|---|
| `path` *(optional)* | A project root, or a page / group folder inside `.lerret/`. Omitted → walks up from CWD to find the nearest project, then exports the whole project. |

When `path` points at a page or group folder, **only artboards within that scope** are captured.

### Options

| Flag | Type | Default | Description |
|---|---|---|---|
| `--format <fmt>` | `png` \| `jpg` | `png` | Output image format. `jpeg` is accepted as an alias for `jpg`. |
| `--out <dir>` | path | `./lerret-export` | Output directory, resolved relative to CWD. Created if absent. |
| `--flat` | boolean | false | Write all images directly under `--out` with collision-disambiguated names. Default: nested folders mirroring page/group hierarchy. |
| `--data <path>` | path | — | JSON or `.js` file whose contents override the data tier (tier 1 of [prop resolution](/concepts/#four-tier-prop-resolution)) for every artboard in this run. Missing / invalid → exit `1`. |
| `--config <path>` | path | — | JSON or `.js` file deep-merged into the cascaded config for this run. Missing / invalid → exit `1`. |
| `-h, --help` | — | — | Print this help and exit `0`. |

### Output structure

**Default (nested):**

```
<out>/
└── <page>/
    └── <group>/
        └── ...
            ├── <asset.name>.<ext>
            └── <asset.name>-<variant>.<ext>
```

**`--flat`:**

```
<out>/
├── <asset.name>.<ext>
└── <asset.name>-<variant>.<ext>
```

In flat mode, file-name collisions (assets with the same `name` in different folders) are disambiguated by joining the path location segments with `-`.

### Override flags: `--data` and `--config`

The override flags let you re-render an entire project with different data or theming **without modifying any files**:

- **`--data <path>`** — a JSON or `.js` file. Its contents override tier 1 (DATA) of prop resolution for every artboard. The file is loaded once before Vite starts; a missing or unparseable file aborts the run immediately (exit `1`) so you get feedback before spending time booting Chromium.
- **`--config <path>`** — a JSON or `.js` file. Its contents are deep-merged into the cascaded config (using the same merge semantics as the on-disk cascade) for every folder in this run.

Neither override is ever written back to your `.lerret/` (the separation invariant). The values flow through the project plugin as in-memory constructor options and are discarded on exit.

A common pattern: keep a `theme-dark.json` and a `theme-light.json` alongside your project, then export both themes with one command each.

### Failure isolation

A failure to capture an **individual artboard** is reported but does **not** abort the run — the remaining artboards still write to disk. The final summary lists per-artboard failures and any custom fonts that could not be embedded across the run.

The run exits `1` (fatal) only for these conditions:

- No project resolved.
- No artboards in scope.
- Output directory could not be created.
- Vite failed to start.
- No Chromium could be launched (neither a system `chrome` / `msedge` channel nor a bundled Playwright browser).
- `--data` or `--config` file missing or unparseable.

### Chromium resolution

`@lerret/cli export` tries a system Chromium **channel** first (`chrome`, then `msedge`) so a zero-install `npx @lerret/cli@latest export` stays light. It falls back to a bundled Playwright browser if one has been installed (`npx playwright install chromium`). If neither is available, the command exits with code `1` and a clear, actionable message about how to install one.

### Examples

```bash
# Whole project, default output
@lerret/cli export

# JPG output to a custom directory
@lerret/cli export --format jpg --out ./assets

# Only the `social` page, flat output
@lerret/cli export .lerret/social --flat

# Re-render with a dark-theme override
@lerret/cli export --config ./theme-dark.json --out ./dark

# Re-render with both data and config overrides (no source edit)
@lerret/cli export \
  --data ./fixtures/release-1.0.json \
  --config ./theme-corporate.json \
  --out ./marketing/release-1.0
```

## `@lerret/cli clear`

Remove sample assets or whole subtrees from a project — the quickest way to wipe scaffolded samples and start fresh.

```bash
@lerret/cli clear [path...] [options]
```

### Arguments

| Argument | Description |
|---|---|
| `path...` | One or more files or folders to remove. Each must be inside the project's `.lerret/` tree. Mutually exclusive with `--all`. |

### Options

| Flag | Description |
|---|---|
| `--all` | Remove every child of `.lerret/` **except** the always-protected `config.json` and `_fonts/`. |
| `--yes` | Skip the confirmation prompt (required for scripts / CI). |
| `--dry-run` | Print the removal plan; do not touch the filesystem. |
| `-h, --help` | Show help. |

Protected paths are never removed: `.lerret/config.json` and `.lerret/_fonts/`. Like every Lerret command, `clear` refuses to touch anything outside `.lerret/`.

## `create-lerret`

Scaffolder for new Lerret projects — `create-*` is the npm convention for scaffolders.

<PM cmd={`npx create-lerret@latest <project-name>                    # five-page teaching preset (default)
npx create-lerret@latest <project-name> --no-samples        # minimal empty project
npx create-lerret@latest <project-name> --preset producthunt # themed starter kit`} />

### Arguments

| Argument | Required | Description |
|---|---|---|
| `<project-name>` | yes | Directory name for the new project. Letters, digits, and `.` `-` `_` only — no path separators, no path traversal. |

### Flags

| Flag | Description |
|---|---|
| `--no-samples` | Scaffold only `.lerret/config.json` (with `{ "vars": {} }`). No `_fonts/`, no sample pages, no sample artboards. The studio opens with a calm empty-but-correct canvas. |
| `--preset <name>` | Scaffold a themed starter kit instead of the teaching preset. Valid: `acme`, `appstore`, `producthunt`, `social-media`, `talks`, `personal`, `live`. Unknown names error with the valid list. |
| `--demo` | Scaffold the default teaching preset and immediately start the dev server with the first-run walkthrough. Mutually exclusive with `--preset` and `--no-samples`. |
| `--no-ai-rules` / `--ai-tools=<list>` | Control which AI-tool integration files ship — see [Editing with AI Tools](/ai-editing/#choosing-which-ai-tools). |

### Default output

The default scaffold is the five-page teaching preset — each page demonstrates one Lerret concept:

```
<project-name>/
└── .lerret/
    ├── README.md
    ├── config.json
    ├── _fonts/
    │   └── LerretFixtureMono.woff2
    ├── intro/      # welcome note (excluded from export)
    ├── landing/    # cascading vars — landing-hero.jsx
    ├── social/     # data files + variants — og-card.jsx, tw-banner.jsx
    ├── brand/      # props schema + validation — business-card.jsx
    └── live/       # auto-refresh — clock.jsx, counter.jsx
```

### Behavior on conflicts

| Condition | Behavior | Exit code |
|---|---|---|
| Target dir does not exist | Create it, populate it. | `0` |
| Target dir is empty | Populate it. | `0` |
| Target dir is non-empty | Refuse with clear message. No writes. | `1` |
| Target path is a file | Refuse. No writes. | `1` |
| Parent dir missing or unwritable | Refuse with clear message. | `1` |
| Mid-copy filesystem failure | Clean up any partial output, then exit. | `1` |

### Network and offline

`create-lerret` itself has **zero runtime dependencies** — the package contains the template and a single Node script. The npm runner (`npx` / `pnpm dlx` / etc.) needs to fetch the package the first time; once cached, scaffolding is offline-capable.

The follow-up `npx @lerret/cli@latest dev` does need network to fetch Vite, React, and `@lerret/cli` the first time, but **never** makes outbound requests at runtime. After the first run, dev startup is offline-capable.
